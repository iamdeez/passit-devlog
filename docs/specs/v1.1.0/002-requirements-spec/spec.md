# Spec: requirements-spec

> Branch: 002-requirements-spec | Date: 2026-06-19 | Version: v1.1.0

## 배경 및 목적

passit-devlog는 공연·스포츠 티켓 실시간 중개 플랫폼 **Passit**을 포트폴리오 공개용으로 재현한 프로젝트다.
001-source-integration 스펙을 통해 원본 레포(CLD4-T2-IAMConan/Passit)의 소스코드를 passit-devlog로 이전했으며,
본 스펙은 이전된 시스템이 충족해야 할 **기능·비기능 요구사항 전체를 공식화**하여 추적 가능한 형태로 정의한다.

이 문서를 기준으로:
- 각 서비스(service-account / service-ticket / service-trade / service-chat / service-cs)의 구현 완료 여부를 검증한다.
- 프론트엔드 화면 흐름이 정의된 유스케이스를 충족하는지 확인한다.
- 비기능 요구사항(성능·보안·가용성 등)에 대한 설계 의사결정 근거를 남긴다.

## 사용자 스토리 (User Stories)

### 구매자 (Buyer)

**회원 & 계정**
- **US-B01** (M): 구매자로서, 플랫폼을 이용하기 위해 회원 가입을 하고 싶다.
- **US-B02** (S): 구매자로서, 이메일 인증을 통해 중복 가입이 방지된 신뢰할 수 있는 계정을 보유하고 싶다.

**티켓 탐색 / 검색**
- **US-B03** (M): 구매자로서, 공연·경기 티켓을 카테고리·키워드 기준으로 조회하고 싶다.
- **US-B04** (C): 구매자로서, 임박 공연(오늘/내일) 기준 잔여 양도표를 빠르게 보고 싶다.

**관심 티켓 관리**
- **US-B05** (S): 구매자로서, 나중에 쉽게 찾을 수 있도록 관심 티켓을 저장(즐겨찾기)하고 싶다.
- **US-B06** (C): 구매자로서, 이전 구매 이력·관심 공연 기반의 맞춤형 티켓 추천을 받고 싶다.

**양도 요청 / 거래**
- **US-B07** (M): 구매자로서, 선택한 티켓에 양도 요청을 보내고 싶다.
- **US-B08** (C): 구매자로서, 결제 후 에스크로로 금액이 잠겨 있어 판매자 먹튀를 걱정하지 않고 싶다.
- **US-B09** (C): 구매자로서, 거래 완료 후 영수증 또는 거래 내역서를 발급받고 싶다.

**안전 / 신고**
- **US-B10** (S): 구매자로서, 좌석 정보가 검증되어 조작된 스크린샷으로부터 보호받고 싶다.
- **US-B11** (M): 구매자로서, 정가 이상의 표를 등록한 판매자를 신고할 수 있고 싶다.

**커뮤니케이션**
- **US-B12** (M): 구매자로서, 판매자와 실시간 채팅을 할 수 있고 싶다.
- **US-B13** (S): 구매자로서, 고객센터에 문의를 남기고 처리 상태를 확인하고 싶다.

**알림**
- **US-B14** (C): 구매자로서, 구매한 공연이 임박했을 때(D-1) 리마인드 알림을 받고 싶다.
- **US-B15** (M): 구매자로서, 양도 요청이 만료·거절될 경우 알림을 받아 시간 낭비를 줄이고 싶다.

**후기 / 평점**
- **US-B16** (S): 구매자로서, 거래 완료 후 판매자에 대한 후기와 별점을 남기고 싶다.

### 판매자 (Seller)

**회원 & 계정**
- **US-S01** (M): 판매자로서, 회원가입 및 이메일 인증을 통해 중복 가입이 방지된 계정으로 플랫폼을 이용하고 싶다.

**티켓 등록 / 관리**
- **US-S02** (M): 판매자로서, 보유 티켓을 플랫폼에 등록하여 빠르게 구매자를 찾고 싶다.
- **US-S03** (M): 판매자로서, 중복으로 같은 티켓을 등록할 수 없어서 실수로 인한 중복 양도를 방지하고 싶다.
- **US-S04** (M): 판매자로서, 등록한 티켓을 조회·수정·삭제할 수 있고 싶다.
- **US-S05** (M): 판매자로서, 정가 검증을 위해 공연명/가격이 적힌 티켓 이미지를 업로드하고 싶다.
- **US-S06** (C): 판매자로서, 티켓 자동 인식(OCR)으로 복잡한 입력 없이 빠르게 등록하고 싶다.
- **US-S07** (C): 판매자로서, 동일 구역·연석 티켓을 묶음 거래할 수 있고 싶다.

**거래 관리**
- **US-S08** (M): 판매자로서, 양도 요청이 들어오면 즉시 알림을 받고 수락 또는 거절하고 싶다.
- **US-S09** (S): 판매자로서, 정가 이상 양도 시도 시 금지 조항 안내를 받아 제재를 예방하고 싶다.

**신뢰 / 후기**
- **US-S10** (S): 판매자로서, 과거 거래 이력·신뢰 점수를 확인·관리하여 구매자에게 신뢰를 줄 수 있고 싶다.

### 관리자 (Admin)

**회원 관리**
- **US-A01** (M): 관리자로서, 회원 정보를 조회·수정·제재(정지/영구정지)할 수 있고 싶다.

**티켓 / 거래 관리**
- **US-A02** (M): 관리자로서, 거래 완료 처리(입장 확인 또는 3일 후 자동 확정)를 할 수 있고 싶다.
- **US-A03** (M): 관리자로서, 공연 취소·변경 시 쌍방 손해 없이 거래를 취소할 수 있고 싶다.
- **US-A04** (S): 관리자로서, 티켓을 대·중·소 카테고리로 구분하여 관리하고 싶다.
- **US-A05** (M): 관리자로서, 공연 시간이 지난 만료 티켓을 삭제 처리할 수 있고 싶다.
- **US-A06** (M): 관리자로서, 인기 공연·티켓 리스트를 관리하여 사용자 검색 편의성을 높이고 싶다.

**신고 / CS 처리**
- **US-A07** (M): 관리자로서, 사용자 신고를 확인하고 경고·정지·무효 처리를 할 수 있고 싶다.
- **US-A08** (C): 관리자로서, 채팅 내 정가 초과 거래 시도를 모니터링할 수 있고 싶다.
- **US-A09** (S): 관리자로서, 특정 판매자의 반복 정가 초과 시 자동 경고·제재 로그를 확인하고 싶다.
- **US-A10** (S): 관리자로서, 채팅 내 위험 키워드(계좌번호, 카카오톡 등) 필터링 규칙을 설정하고 싶다.

**콘텐츠 / 공지**
- **US-A11** (M): 관리자로서, 공지사항을 작성하여 플랫폼 회원에게 정보를 전달하고 싶다.
- **US-A12** (S): 관리자로서, FAQ를 관리하고 카테고리별로 분류하고 싶다.
- **US-A13** (M): 관리자로서, 사용자 1:1 문의에 답변할 수 있고 싶다.

**통계 / 대시보드**
- **US-A14** (C): 관리자로서, 전체 거래 건수·거래액 등을 실시간으로 확인하고 싶다.
- **US-A15** (S): 관리자로서, 월별 거래량·취소율·신고율 통계를 조회하여 운영 정책을 개선하고 싶다.

## 기능 요구사항 (Functional Requirements)

### 인증 & 계정 (Auth & Account)

- **FR-001** (M): 시스템은 이메일·비밀번호 기반 회원가입을 제공해야 한다.
- **FR-002** (S): 시스템은 회원가입 시 이메일 인증을 통해 중복 가입을 방지해야 한다.
- **FR-003** (M): 시스템은 JWT 기반 로그인 및 Refresh Token Rotation을 지원해야 한다.
- **FR-004** (S): 시스템은 카카오 OAuth 소셜 로그인을 지원해야 한다.
- **FR-005** (M): 사용자는 이메일·닉네임·비밀번호·프로필 사진 등 계정 정보를 수정할 수 있어야 한다.
- **FR-006** (M): 사용자는 비밀번호를 분실했을 경우 이메일을 통해 재설정할 수 있어야 한다.

### 티켓 탐색 (Ticket Browse)

- **FR-007** (M): 사용자는 카테고리·키워드·상태 필터로 티켓 목록을 페이징 조회할 수 있어야 한다.
- **FR-008** (M): 사용자는 티켓 상세 정보(공연명, 날짜, 좌석, 가격, 이미지 등)를 조회할 수 있어야 한다.
- **FR-009** (C): 시스템은 임박 공연(오늘/내일) 기준으로 잔여 티켓을 별도 노출해야 한다.
- **FR-010** (C): 시스템은 사용자 관심 이력 기반의 맞춤형 티켓을 추천해야 한다.

### 티켓 등록 & 관리 (Ticket Management — Seller)

- **FR-011** (M): 판매자는 이미지(S3 업로드)와 함께 티켓을 등록할 수 있어야 한다.
- **FR-012** (M): 시스템은 동일 판매자의 중복 티켓 등록을 방지해야 한다.
- **FR-013** (M): 판매자는 자신이 등록한 티켓을 수정·삭제할 수 있어야 한다.
- **FR-014** (M): 관리자는 공연 시간이 지난 만료 티켓을 삭제 처리할 수 있어야 한다.
- **FR-015** (S): 판매자가 정가 초과 가격을 입력하려 할 경우 시스템이 경고 메시지를 표시해야 한다.

### 양도 거래 (Trading & Deal)

- **FR-016** (M): 구매자는 원하는 티켓에 양도 요청(Deal)을 생성할 수 있어야 한다.
- **FR-017** (M): 판매자는 수신된 양도 요청을 수락 또는 거절할 수 있어야 한다.
- **FR-018** (M): 구매자는 판매자가 수락한 거래에 대해 결제를 진행할 수 있어야 한다.
- **FR-019** (M): 거래 상태는 요청 → 수락/거절 → 결제 → 완료 단계로 추적 가능해야 한다.
- **FR-020** (M): 관리자는 거래를 강제 완료 또는 강제 취소 처리할 수 있어야 한다.
- **FR-021** (M): 구매자와 판매자는 자신의 거래 목록을 상태별로 조회할 수 있어야 한다.
- **FR-022** (C): 시스템은 결제 금액을 에스크로 방식으로 보관하여 사기를 방지해야 한다.

### 실시간 채팅 (Chat)

- **FR-023** (M): 구매자는 판매자와 1:1 채팅방을 개설할 수 있어야 한다.
- **FR-024** (M): 채팅방 내에서 양도 요청·수락·거절 액션을 메시지 형태로 주고받을 수 있어야 한다.
- **FR-025** (M): 사용자는 자신의 채팅 목록과 채팅 이력을 조회할 수 있어야 한다.
- **FR-026** (S): 채팅 내 위험 키워드(계좌번호, 타 메신저 유도 등)는 자동으로 필터링되어야 한다.
- **FR-027** (C): 관리자는 특정 채팅방 대화 내용을 모니터링할 수 있어야 한다.

### 고객지원 (Customer Support)

- **FR-028** (M): 사용자는 신고를 접수할 수 있어야 한다.
- **FR-029** (M): 관리자는 신고 목록을 조회하고 경고·정지·무효 처리를 할 수 있어야 한다.
- **FR-030** (S): 사용자는 1:1 문의를 등록하고 처리 상태를 확인할 수 있어야 한다.
- **FR-031** (M): 관리자는 사용자 문의에 답변할 수 있어야 한다.
- **FR-032** (M): 관리자는 공지사항을 작성·수정·삭제할 수 있어야 한다.
- **FR-033** (S): 관리자는 FAQ를 작성·수정·삭제하고 카테고리로 분류할 수 있어야 한다.

### 회원 관리 (Admin — User)

- **FR-034** (M): 관리자는 전체 회원 목록을 조회하고 이메일·닉네임·상태로 필터링할 수 있어야 한다.
- **FR-035** (M): 관리자는 회원 상태를 정상·정지·영구정지로 변경할 수 있어야 한다.

### 카테고리 & 콘텐츠 관리 (Admin — Category & Content)

- **FR-036** (S): 관리자는 공연·스포츠 대·중·소 카테고리를 생성·수정·삭제할 수 있어야 한다.

### 후기 & 평점 (Review)

- **FR-037** (S): 구매자는 거래 완료 후 판매자에 대한 후기와 별점을 등록할 수 있어야 한다.
- **FR-038** (S): 사용자는 마이페이지에서 자신이 받은 후기와 별점을 조회할 수 있어야 한다.

### 알림 (Notification)

- **FR-039** (M): 양도 요청 만료·거절 시 구매자에게 알림이 발송되어야 한다.
- **FR-040** (M): 양도 요청 수신 시 판매자에게 알림이 발송되어야 한다.
- **FR-041** (C): 공연 D-1일에 해당 공연 구매자에게 리마인드 알림이 발송되어야 한다.

## 비기능 요구사항 (Non-Functional Requirements)

### 성능

- **NFR-001** (M): 피크 시간 동시 접속자 1,000 CCU / 일반 200 CCU를 처리할 수 있어야 한다.
- **NFR-002** (M): 주요 API(로그인·티켓 조회·양도 요청) P95 응답시간 700ms 이하를 유지해야 한다.
- **NFR-003** (M): 공연·티켓 목록 조회 P95 응답시간 1,000ms 이하를 유지해야 한다.
- **NFR-004** (M): 알림·로그·이미지 처리 등 비동기 작업은 API 응답시간에 영향을 줘서는 안 된다.
- **NFR-005** (S): 채팅 내 금칙어 감지는 500ms 이내에 판별하여 블라인드 처리해야 한다.

### 가용성

- **NFR-006** (M): 핵심 서비스(로그인·티켓 조회·거래·알림)는 개별 서비스 장애 시에도 독립적으로 가용해야 한다.
- **NFR-007** (M): 공연 시작 전후 3시간 및 티켓 오픈 당일은 배포 금지(Code Freeze) 및 무중단 운영을 보장해야 한다.

### 보안

- **NFR-008** (M): 개인정보(이름·전화번호)는 DB 저장 시 암호화되어야 한다(비밀번호 bcrypt, 식별값 AES-256).
- **NFR-009** (M): JWT Refresh Token Rotation을 적용하여 토큰 탈취 리스크를 최소화해야 한다.
- **NFR-010** (M): 모든 전송 구간은 TLS로 암호화되어야 한다.
- **NFR-011** (M): 역할 기반 권한 구분(구매자 / 판매자 / 관리자)을 적용해야 한다.
- **NFR-012** (M): 이메일 인증을 통해 중복 가입을 방지해야 한다.
- **NFR-013** (M): Secrets는 로컬에서 `.env` 파일로, 운영에서 AWS Secret Manager로 관리해야 한다.
- **NFR-014** (C): 관리자 API는 IP 제한 또는 2FA를 적용해야 한다.

### 확장성

- **NFR-015** (M): 서비스는 MSA 구조로 기능별 독립 확장이 가능해야 한다.
- **NFR-016** (M): 웹 서버 CPU 사용량 70% 도달 시 Auto Scaling이 트리거되어야 한다.
- **NFR-017** (S): DB는 읽기 전용 복제본을 2개 이상 두어 조회 트래픽을 분산해야 한다.
- **NFR-018** (C): 공연·티켓 목록은 Redis 캐시를 통해 응답 성능을 개선해야 한다.

### 재해복구 (DR)

- **NFR-019** (M): 데이터 손실 허용(RPO) 10분 이내, 복구 목표(RTO) 4시간 이내를 달성해야 한다.

### 신뢰성

- **NFR-020** (M): 티켓 거래·양도 상태 변경은 DB 트랜잭션 무결성을 보장해야 한다.
- **NFR-021** (M): 동일 티켓에 복수 구매자가 동시에 양도 요청·결제를 시도할 경우 1명만 성공해야 한다(동시성 제어).
- **NFR-022** (M): 티켓 양도 가능 여부 및 상태는 실시간으로 정확하게 동기화되어야 한다.

### 모니터링 & 로깅

- **NFR-023** (M): 서버 CPU·메모리·트래픽은 실시간으로 모니터링되어야 하며, 임계치 초과 시 관리자에게 알림이 발송되어야 한다.
- **NFR-024** (M): 거래 로그(요청·수락·취소·완료)와 관리자 액션 로그가 기록되어야 한다.

### 유지보수성

- **NFR-025** (M): 모든 서비스는 Controller / Service / Domain / Repository 레이어를 준수해야 한다.
- **NFR-026** (S): API 명세는 Swagger 또는 동등 도구로 자동 문서화되어야 한다.

### UX

- **NFR-027** (M): 티켓 검색부터 양도 완료까지 모든 과정이 직관적인 화면 흐름으로 제공되어야 한다.
- **NFR-028** (M): 로딩이 길어질 경우 Skeleton UI 또는 로딩 프로그레스바를 표시해야 한다.
- **NFR-029** (S): 에러·예외 상황에 대해 사용자 친화적인 메시지를 제공해야 한다.

### DevOps

- **NFR-030** (M): CI/CD 파이프라인을 통해 코드 push 시 자동 빌드·배포되어야 한다.
- **NFR-031** (M): 개발 / 운영 환경이 분리되어야 한다.
- **NFR-032** (M): 인프라 구성은 Terraform으로 코드 관리되어야 한다.
- **NFR-033** (S): GitOps(ArgoCD) 기반 운영 자동화가 적용되어야 한다.

## 수용 기준 (Acceptance Criteria)

### 인증 & 계정

- **SC-001** (`FR-001`): 이메일·비밀번호로 회원가입 요청 시 계정이 생성되고 200 응답이 반환된다.
- **SC-002** (`FR-002`): 동일 이메일로 재가입 시도 시 409 Conflict가 반환되며, 이메일 미인증 상태에서는 주요 API가 차단된다.
- **SC-003** (`FR-003`): 로그인 성공 시 Access Token과 Refresh Token이 발급되고, Refresh 요청 시 기존 Refresh Token이 무효화되며 새 토큰 쌍이 반환된다.
- **SC-004** (`FR-004`): 카카오 OAuth 콜백 처리 시 신규 사용자는 가입, 기존 사용자는 로그인 처리된다.
- **SC-005** (`FR-005`): 닉네임·프로필 사진 수정 요청이 DB에 반영되며 변경된 값이 조회 API에서 확인된다.
- **SC-006** (`FR-006`): 비밀번호 재설정 이메일 발송 후 링크로 신규 비밀번호 설정이 가능하다.

### 티켓

- **SC-007** (`FR-007`): 카테고리 ID·키워드·상태 파라미터로 티켓 목록 조회 시 필터링된 결과가 페이지네이션으로 반환된다.
- **SC-008** (`FR-008`): 티켓 ID로 상세 조회 시 이미지 URL, 공연명, 날짜, 좌석, 가격이 포함된 응답이 반환된다.
- **SC-009** (`FR-011`): S3 이미지 업로드 URL과 함께 티켓 등록이 완료되면 해당 티켓이 목록에 노출된다.
- **SC-010** (`FR-012`): 동일 판매자가 이미 등록한 티켓과 동일 좌석·날짜로 재등록 시도 시 400 에러가 반환된다.
- **SC-011** (`FR-013`): 판매자 본인만 자신의 티켓을 수정·삭제할 수 있고, 타인 접근 시 403이 반환된다.
- **SC-012** (`FR-014`): 관리자 API 호출로 만료 티켓 삭제 처리 시 해당 티켓의 status가 변경되거나 목록에서 제외된다.

### 거래

- **SC-013** (`FR-016`): 구매자가 티켓 ID로 양도 요청 시 Deal이 생성되고 PENDING 상태로 반환된다.
- **SC-014** (`FR-017`): 판매자가 수락 시 Deal 상태가 ACCEPTED로 변경되고, 거절 시 REJECTED로 변경된다.
- **SC-015** (`FR-018`): 결제 API 호출 성공 시 Deal 상태가 PAID로 변경된다.
- **SC-016** (`FR-019`): 거래 상태 이력(PENDING→ACCEPTED→PAID→COMPLETED)이 API로 조회 가능하다.
- **SC-017** (`FR-020`): 관리자 강제 완료/취소 API 호출 시 Deal 상태가 즉시 변경되고 양측에 알림이 발송된다.
- **SC-018** (`FR-021`): 구매자·판매자가 상태 파라미터로 자신의 Deal 목록을 조회할 수 있다.

### 채팅

- **SC-019** (`FR-023`): 채팅방 생성 API 호출 시 고유한 Room ID가 반환된다.
- **SC-020** (`FR-024`): WebSocket/STOMP 연결 후 메시지 송신 시 상대방에게 실시간으로 전달된다.
- **SC-021** (`FR-024`): 채팅 내 양도 요청 메시지 발송 시 시스템 액션 메시지로 처리되고, 수락·거절 버튼이 표시된다.
- **SC-022** (`FR-025`): 채팅방 목록 API가 최신 메시지·읽음 여부를 포함하여 반환된다.

### 고객지원

- **SC-023** (`FR-028`): 신고 접수 시 신고 ID가 반환되고 관리자 목록에 노출된다.
- **SC-024** (`FR-029`): 관리자가 경고/정지/무효 처리 시 신고 status가 업데이트되고 신고자에게 알림이 발송된다.
- **SC-025** (`FR-030`): 문의 등록 시 status가 PENDING이 되고, 사용자는 자신의 문의 목록을 조회할 수 있다.
- **SC-026** (`FR-031`): 관리자가 답변 등록 시 status가 ANSWERED로 변경된다.
- **SC-027** (`FR-032`): 관리자가 공지사항 생성 시 사용자 공지 목록에 즉시 노출된다.

### 회원 관리 (Admin)

- **SC-028** (`FR-034`): 관리자 회원 목록 API가 이메일·닉네임·상태 필터 파라미터를 지원한다.
- **SC-029** (`FR-035`): 관리자의 회원 상태 변경 API 호출 시 해당 사용자의 다음 요청에서 401/403이 반환된다.

### 후기

- **SC-030** (`FR-037`): 거래 COMPLETED 상태에서만 리뷰 등록이 가능하고, 중복 리뷰 시 409가 반환된다.
- **SC-031** (`FR-038`): 마이페이지 활동 이력 API에서 수신한 후기·평균 별점이 조회된다.

## 화면 흐름 요약

### 판매자 흐름
```
메인 → 회원가입/로그인 → 티켓 등록 → (마이페이지 등록 티켓 확인)
→ 구매자 채팅 → 양도 수락/거절 → 결제 확인 → 거래 완료
```

### 구매자 흐름
```
메인 → 회원가입/로그인 → 티켓 검색/조회 → 상세 → 채팅 개설
→ 양도 요청 → 결제 → 거래 완료 → 후기/별점 등록
→ 마이페이지에서 구매 내역 확인
```

### 관리자 흐름
```
관리자 로그인 → 대시보드
├── 회원 관리: 목록 조회 → 상세 → 상태 변경
├── 거래 관리: 목록(상태 필터) → 상세 → 강제 완료/취소
├── 신고 관리: 목록 → 상세 → 경고/정지/무효 처리
├── 카테고리 관리: 트리 확인 → 생성/수정/삭제
└── 콘텐츠 관리: 공지/FAQ/문의/가이드 CRUD
```
